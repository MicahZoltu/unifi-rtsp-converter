# 06 — Service account least-privilege

## Goal

Stop running the Windows service as `LocalSystem` (the SCM default when `start_name = null`). Run it under a least-privilege account that can still write to the user-owned exe directory (so cert generation, the log, and any state beside the exe keep working without relocating them to `%PROGRAMDATA%`).

## Context

`service.rs:361-362` passes `start_name = null` to `CreateServiceW`, which selects `LocalSystem` — full machine privilege, more than a streaming proxy needs. The original DEBT.md plan was to switch to `LocalService` and relocate the log/cert to `%PROGRAMDATA%\flvproxy` (because `LocalService` cannot write to `Program Files`). The revised deployment model (exe lives in a user-owned directory, service runs as the installing user) makes a simpler, no-relocation approach possible.

## Recommended approach: `NT SERVICE\flvproxy` virtual account

Windows offers *per-service virtual accounts* of the form `NT SERVICE\<ServiceName>`. A service configured to run as `NT SERVICE\flvproxy` gets its own low-privilege virtual account that:

- Needs **no password** — `CreateServiceW` with `start_name = "NT AUTHORITY\\LocalService"`? No: use `start_name = "NT SERVICE\\flvproxy"`, `password = null`. The SCM grants the account `SeServiceLogonRight` automatically and it is created on first start.
- Is least-privilege by default (no admin token, no network credential, unlike `LocalSystem`).
- Is **distinct from `LocalService`**: it is a dedicated per-service SID, not the shared `LocalService` account, so it can be granted ACLs on exactly the resources this service needs.
- Can write to the user-owned exe directory once an ACL grants `NT SERVICE\flvproxy` write access there.

This matches the user's intent ("runs as the installing user" → in practice a dedicated low-privilege service identity that owns the exe dir) without the password-prompt complexity of running as a human user account (which `CreateServiceW` would require a password for).

### Steps

1. In `service::win::install`, compute the service account name: `format!("NT SERVICE\\{SERVICE_NAME}")` → `"NT SERVICE\\flvproxy"`. Pass it as `start_name` (wide-encoded) to `CreateServiceW`, with `password = null`. The SCM resolves the virtual account on first start.
2. After `CreateServiceW` succeeds, grant the account write access to the exe directory so the service can write the log and the lazily-generated PFX. Use `GetNamedSecurityInfoW` / `SetEntriesInAclW` / `SetNamedSecurityInfoW` (advapi32) to add an `ACCESS_ALLOWED_ACE` for `NT SERVICE\flvproxy` with `FILE_GENERIC_WRITE | FILE_GENERIC_READ | FILE_LIST_DIRECTORY | FILE_TRAVERSE` to the exe directory's DACL. This is the bulk of the FFI work; mirror the existing advapi32 FFI style in `service::win`. If the ACL grant fails, log a warning and continue — the service will start but may fail to write the log; the operator can fix the dir ACL manually. Do not abort the install over the ACL grant (the cert was already generated by step 05's `--install` path as admin).
3. Update the `CreateServiceW` call's `start_name` doc comment (`service.rs:361`) to describe the `NT SERVICE\flvproxy` virtual account and the dir-ACL grant, and delete the old "see `DEBT.md` for the trigger" sentence.
4. Update the module-level doc (`service.rs:1`) if it mentions `LocalSystem`.
5. The `--uninstall` path does not need to revoke the ACL (leaving a per-service-SID ACE on a user dir is harmless once the service is deleted — the SID becomes unresolvable but inert). Optionally revoke it for cleanliness; skip unless trivial.

## Scope

In: `start_name` change to `NT SERVICE\flvproxy`; advapi32 ACL grant on the exe directory; doc updates.

Out: relocating the log/cert to `%PROGRAMDATA%` (not needed — the service writes beside the exe in a user-owned dir with an explicit ACL grant); prompting for a password; any change to `run_as_service` / `service_main` / the SCM lifecycle callbacks; any change to `App::bootstrap` log path.

## Alternative considered (installing user with password)

Running the service as the literal installing user requires `CreateServiceW` with `start_name = "<DOMAIN>\\<user>"` and `password = <user's password>` — the SCM must have credentials to log the account on as a service. This means prompting for the password at `--install` time (interactive prompt or `--service-user`/`--service-password` flags), which adds CLI surface and password-handling risk (shell history). The `NT SERVICE\flvproxy` virtual account achieves the same least-privilege + writable-exe-dir goal with no password. Use the virtual account unless a future requirement demands the human user's identity specifically.

## Test

- The ACL FFI cannot run on Linux. Verify `cargo build --target x86_64-pc-windows-gnu` compiles the advapi32 calls and the `start_name` change.
- Add a Linux-visible test (in `service.rs` top-level `#[cfg(test)]`) that constructs the account-name string `"NT SERVICE\\flvproxy"` from `SERVICE_NAME` and asserts it equals `format!("NT SERVICE\\{SERVICE_NAME}")` — guards the formatting.
- Manual Windows smoke test (noted in acceptance): `--install`, `sc start flvproxy`, confirm the service runs as `NT SERVICE\flvproxy` (visible in `sc qc flvproxy` `SERVICE_START_NAME`), confirm `flvproxy.log` is writable, `sc stop`, `--uninstall`.

## Acceptance

- `sc qc flvproxy` shows `SERVICE_START_NAME: NT SERVICE\flvproxy` after `--install`.
- The service starts and writes `flvproxy.log` beside the exe (ACL grant worked).
- `--uninstall` still stops and deletes the service cleanly.

- No remaining `DEBT.md` reference in `service.rs`.

## Notes

- This step shares the `--install` entry point with step 05 (cert generation). Both modify `win::install`; coordinate so the cert generation (step 05) lands first and this step's ACL grant runs after cert generation in the same function. The two changes do not conflict semantically.
- If the ACL-grant FFI proves large/risky, the acceptable fallback is: change `start_name` to `NT SERVICE\flvproxy` and print an instruction telling the operator to `icacls <dir> /grant "NT SERVICE\flvproxy:(OI)(CI)W"` manually. Prefer the automatic grant, but do not block the step on it — a printed instruction is better than `LocalSystem`.
