# Technical Debt Ledger

This file is the single source of truth for deferred work, known hacks, and "good enough for now" decisions. It is maintained from build-plan step 00 onward. See `plan/README.md` → "Quality Bar & Anti-Debt Discipline" for the rules.

## Format

One line per item:

```
step NN | <file>:<area> | <what is the debt> | <FIX NOW | TRIGGER: concrete future event>
```

- `FIX NOW` items **must** be resolved before the next dedicated review milestone (`07`, `13`, `25`, `28`).
- `TRIGGER:` items name the exact future event that forces revisiting them.
- At each review, reconcile: every item is either resolved (delete the line) or re-justified with a fresh trigger. No item lives forever unchallenged.
- If this file is empty, that is the goal state — say so explicitly in each review ("DEBT.md empty: confirmed").
- **Resolved items are deleted** (history lives in git), not annotated as "RESOLVED" and retained. A line present here means the debt is still open; "resolution records" are an anti-pattern that hides whether the ledger reflects real pending work.

## Active items

step 00 | .cargo/config.toml:rustflags | Dropped `-static-libwinpthread` from the Windows GNU static-link rustflags: this build host's MinGW-w64 GCC 14 uses the win32 thread model, which rejects that flag (it is a posix-thread-model flag) and does not link winpthread at all, so the two remaining flags already yield a self-contained exe (verified via objdump). | TRIGGER: build host switches to a posix-thread-model MinGW (`x86_64-w64-mingw32-gcc -v` reports `--enable-default-msvcrt`/posix threads), at which point re-add static winpthread linking to keep the exe free of `libwinpthread-1.dll`.

step 17 | src/tls_schannel.rs:edge cases + write/read hardening | The hand-rolled TLS is validated against exactly one camera (fw 4.73.112) and one .NET `SslStream` client. Edge cases (very large frames, rapid reconnects, partial records at unusual boundaries) are covered by the self-test's 1 B / 64 KiB / 1 MiB round-trips but not exhaustively. Two simplifications remain: (a) `TlsStream::write` sends each encrypted record with a single `stream.write_all` and propagates `WouldBlock`/`TimedOut` as a fatal connection error (the record may be half-sent) rather than buffering+resending like the `schannel` crate — acceptable because the recon/self-test sockets set no write timeout, but production may need resumable writes; (b) the streaming `Read` path propagates `WouldBlock`/`TimedOut` from the underlying socket (no internal retry/deadline) so the caller controls cadence, which is correct but means a malformed-record / peer-reset path is not yet hardened. The step-28 final review examined this and decided not to add resumable write buffering now: the Windows-only 7442 TLS listener is neither compiled nor tested on the Linux build host, the never-crash guarantee for the Linux-testable data path was already covered in step 26 (`catch_unwind` + FLV resync + RTSP bounds), and no production trace has surfaced a half-sent-record or peer-reset crash on 7442 — adding speculative buffering now would be the "just in case" feature the review's "no new features" rule guards against. | TRIGGER: a real-camera capture that exercises a malformed-record / peer-reset path on the 7442 TLS channel (a production trace showing a half-sent TLS record or a peer-reset crash on 7442) — then add resumable write buffering and a malformed-record guard there.

step 22 | src/onvif_server.rs:DEFAULT_FIRMWARE + DEFAULT_SERIAL | `GetDeviceInformation` advertises a fixed firmware (`4.73.112`) and a placeholder serial (`000000000000`) because the proxy does not learn the camera's real firmware/serial from the AVClient exchange or any other channel. `DEFAULT_FIRMWARE` is a sensible UVC G5 value; `DEFAULT_SERIAL` is a non-empty placeholder that keeps `GetDeviceInformation` well-formed. Operators can override firmware via config but serial cannot be learned. The step-25 ONVIF cluster review decided to keep the defaults: the recon'd AVClient exchange (`hello`/`paramAgreement`/`timeSync`) carries no firmware/serial fields, so wiring a shared `Arc<Mutex<CameraIdentity>>` from the Protect controller to `OnvifServer` would have nothing to populate — the static defaults are the intentional design until a capture proves the fields exist. | TRIGGER: a real-camera AVClient capture reveals whether the `hello`/`paramAgreement` exchange carries firmware/serial fields the proxy can forward — if so, wire them through and drop the defaults.

step 24 | src/onvif_discovery.rs:fdPHost coexistence on Windows | The WS-Discovery socket binds `0.0.0.0:3702` with `SO_REUSEADDR` so it coexists with the Windows "Function Discovery Provider Host" service (`svchost.exe` / `fdPHost`), which also binds that address. A specific-IP bind (`192.168.50.100:3702`) to win multicast delivery was attempted but Windows refuses it (`WSAEADDRNOTAVAIL`, os error 10049) when the wildcard is already held. The pktmon capture during the step-24 human test confirmed Probes *do* reach the proxy socket with `fdPHost` running, so no service-stop is required on the tested host; if a future host exhibits dropped Probes (svchost swallowing them), stopping `fdPHost` (`Stop-Service fdPHost`) frees the port and makes the proxy the sole holder. | TRIGGER: a deployment host where WS-Discovery Probes are confirmed (via pktmon) arriving on the NIC but the proxy log shows no `probe from ...` lines — then stop `fdPHost` and retest, and if that resolves it, document the service-stop as a required install step in the `--install` action (future Windows-service step).

step 24 | src/onvif_server.rs:unimplemented ONVIF operations | The proxy implements `GetSystemDateAndTime`, `GetCapabilities`, `GetDeviceInformation`, `GetEndpointReference`, `GetServices`, `GetProfiles`, and `GetStreamUri`. The step-24 Onvier human test showed it also calls `GetAudioOutputConfigurations`, `GetSnapshotUri`, and `SetSynchronizationPoint`, which return a SOAP Fault (`wsa:ActionNotSupported`). Onvier tolerated the faults and completed device-add + live-view, but a stricter NVR might abort on any fault. The step-25 ONVIF cluster review decided to keep the fault responses for now: only Onvier has been tested (it tolerates the faults), so implementing the cheap stubs (`GetSnapshotUri` empty/disabled URI, `GetAudioOutputConfigurations` empty list, `SetSynchronizationPoint` no-op success) is deferred until a real target NVR actually requires them — adding speculative handlers is the "just in case" feature the review's "no new features" rule guards against. | TRIGGER: a target NVR (Blue Iris / Synology) aborts device-add or live-view on the `ActionNotSupported` fault for one of these three operations — then implement the matching stub.

step 24 | src/onvif_discovery.rs:IP_MULTICAST_IF egress not pinned | `bind_multicast_socket` joins the multicast group on the configured `server_ip` interface but cannot pin multicast *egress* to that NIC because std does not expose `IP_MULTICAST_IF`. The one-shot `Hello` on startup therefore leaves via the OS default route for `239.255.255.250`, which on a multi-homed host may be the wrong NIC. The Probe→ProbeMatch flow (unicast reply to the probe sender) routes correctly and is how ONVIF clients discover, so this only affects `Hello`-only listeners. The step-24 human test confirmed Onvier (a Probe-based client) discovers reliably without egress pinning. | TRIGGER: a deployment where a `Hello`-only NVR (one that waits for unsolicited `Hello` rather than probing) misses the device on a multi-homed host — then add raw `ws2_32::setsockopt(IP_MULTICAST_IF)` FFI on Windows to pin egress to the `server_ip` NIC.

step 26 | src/flv_parser.rs:resync only matches standard-layout tag types | The step-26 resync scan matches only tag-type bytes in `{0x08, 0x09, 0x12}` with a standard-layout header, per `plan/26` task 1. UniFi's extendedFlv format uses type-`0x00` video tags with a non-standard swapped header (timestamp before data-size — see `State::TagHeader`), so matching `0x00` would misparse it; the scan therefore skips `0x00` tags. Consequence: a real production 7550 stream (which is extendedFlv / `0x00`) that desyncs cannot be recovered by `resync()` — the buffer grows until `MAX_RESYNC_BUFFER_BYTES` and the connection is dropped (the camera then reconnects, so this is graceful degradation, not a crash). This is acceptable because the production path has never required mid-stream resync in practice (the camera sends clean FLV); the standard-FLV / uPFLV-prefix test path (`0x09` tags) resyncs correctly and is what the step-26 tests exercise. | TRIGGER: a real extendedFlv (`0x00`-tag) stream is observed to desync often enough that the drop-and-reconnect cycle becomes an operational problem — then teach `resync()` the swapped-header layout for `0x00` candidates (decode timestamp-before-dsize, validate the same way) so the production path recovers in place.

step 27 | src/service.rs:install service account | `CreateServiceW` registers the service with `start_name = null` (`LocalSystem`) rather than `NT AUTHORITY\LocalService` because the logger writes `flvproxy.log` beside the exe, which is typically in a directory `LocalService` cannot write to (e.g. under `Program Files`) — `LocalSystem` can write anywhere, avoiding a log-open failure on a default install location. `LocalService` is the safer least-privilege identity and would be preferred once the log (and any other writable state) lives in a `LocalService`-writable location such as `%PROGRAMDATA%\flvproxy`. The Windows-only service FFI cannot be exercised on the Linux build host, so this was validated by compile-check (`cargo check --target x86_64-pc-windows-gnu`) and the step-27 human test is the first runtime confirmation. | TRIGGER: when the log path / writable state is relocated to a `LocalService`-writable directory, switch `CreateServiceW`'s `start_name` to `NT AUTHORITY\LocalService` (wide) and re-run the step-27 human test to confirm the service still starts and writes its log.
